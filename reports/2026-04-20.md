### CLAUDE.md: `/init` コマンドと精度重視の記述原則

**発見内容**: 公式ドキュメントとコミュニティ記事で「精度（Precision）がリビジョン数を削減する」という原則が繰り返し強調された。また `/init` コマンドによる自動生成が新たに推奨手順として明確化。

- **`/init` コマンド**: プロジェクトディレクトリで実行すると、ディレクトリ構造・技術スタックを検出して starter CLAUDE.md を自動生成。ゼロから書く必要がなくなる。
- **具体的参照の原則**: ファイルパス・制約・サンプルパターンを明示するほどリビジョンが減る。「TypeScript を使う」より「`src/api/**/*.ts` のファイルは必ず Zod でバリデーションを行う」のように記述。
- **10-section 構造**: Medium 記事で効果が検証された推奨構造:
  1. Project Context（1行サマリー）
  2. Architecture Overview（ディレクトリ構成）
  3. Tech Stack（バージョン含む）
  4. Code Standards（型ヒント・命名規則等）
  5. Key Commands（test/build/lint 正確なコマンド）
  6. Environment Setup（依存関係の注意点）
  7. Gotchas（既知の罠・回避策）
  8. Compaction Instructions（コンパクション後に保持する情報）
  9. Anti-patterns（禁止パターン）
  10. References（重要ドキュメントへのリンク）

**適用提案**:
```bash
# 既存プロジェクトに即座に適用
cd /path/to/project
claude
/init
# → 生成されたドラフトを上記10-section 構造に沿って補強する
```

### hooks: 3つのカデンス分類と実行順の設計原則

**発見内容**: hooks のイベントは「いつ発火するか」で3カデンスに分類されており、それぞれ設計指針が異なることが公式ドキュメントで明確化された（前回レポートはイベント名の列挙のみ）。

| カデンス | イベント | 設計指針 |
|---------|---------|---------|
| **セッションあたり1回** | `SessionStart`, `SessionEnd` | セットアップ・クリーンアップ処理。重い初期化はここへ |
| **ターンあたり1回** | `UserPromptSubmit`, `Stop`, `StopFailure` | ターン完了後の検証。テストスイート・ビルドはここ |
| **ツール呼び出しごと** | `PreToolUse`, `PostToolUse` | 軽量処理のみ。重い処理は`Stop`へ移行を推奨 |

**新たに判明した実装パターン**:

1. **PreToolUse でのブロック理由を stderr に書く**:
   ```bash
   #!/bin/bash
   # PreToolUse: 機密ファイルへの書き込みをブロック
   if echo "$CLAUDE_TOOL_INPUT" | grep -q "\.env\|secrets/"; then
     echo "機密ファイルへの書き込みはブロックされました: $CLAUDE_TOOL_INPUT" >&2
     exit 2  # exit 2 = ブロック。理由は stderr 経由で Claude に伝達
   fi
   ```

2. **PostToolUse の `; exit 0` フォールバック**:
   ```bash
   #!/bin/bash
   # PostToolUse: 自動フォーマット（失敗してもワークフローをブロックしない）
   prettier --write "$CLAUDE_TOOL_OUTPUT_PATH" 2>/dev/null; exit 0
   ```

3. **カデンス別の責務分離パターン**:
   ```json
   {
     "hooks": {
       "PreToolUse": [
         {"matcher": "Write|Edit", "command": "~/.claude/hooks/scan-secrets.sh"}
       ],
       "PostToolUse": [
         {"matcher": "Write|Edit", "command": "prettier --write $FILE; exit 0"}
       ],
       "Stop": [
         {"command": "npm test && npm run lint || true"}
       ],
       "SessionStart": [
         {"command": "~/.claude/hooks/restore-context.sh"}
       ]
     }
   }
   ```

**適用提案**: 既存の PostToolUse にテストスイートを置いている場合、`Stop` フックに移行してツール呼び出しのたびに実行されるオーバーヘッドを排除する。

### MCP: `managed-mcp.json` によるエンタープライズ配布

**発見内容**: チーム/エンタープライズ環境向けに `managed-mcp.json` でサーバー設定を固定配布できることが判明。ユーザーによるカスタマイズを許可しつつポリシー制御を維持できる。

**設定の使い分け**:
| ファイル | 用途 | カスタマイズ |
|---------|------|------------|
| `.mcp.json` | プロジェクト共有（git 管理） | チームメンバーが変更可 |
| `managed-mcp.json` | 組織ポリシー固定 | ユーザーは追加のみ可（既存を変更不可） |
| `~/.claude/mcp.json` | 個人グローバル設定 | 完全自由 |

**セキュリティベストプラクティス（最小権限原則）**:
```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN_READONLY}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres", "${DB_URL_READONLY}"]
    }
  }
}
```

**適用提案**: 
- `GITHUB_TOKEN` は `repo:read`・`issues:read` のみのスコープで発行し `${ENV_VAR}` 参照に。
- Tool Search 機能（MCP ツール定義を遅延ロード）により大量ツールがあってもコンテキストを 85〜95% 削減。Context7 MCP でドキュメント参照を強化する場合も同様。

### Auto-memory: Claude が自律的に知識を蓄積する仕組み

**発見内容**: Claude Code v2.1.59 以降に追加された Auto-memory システムの詳細が公式ドキュメントで確認された。過去のレポートでは言及のみだったが、実用的な設定・運用方法が明確化。

**ストレージ構造**:
```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # インデックス（セッション開始時に最大200行/25KB 読み込み）
├── debugging.md       # デバッグパターン（オンデマンドで読み込み）
├── api-conventions.md # API 設計決定事項
└── ...                # Claude が自律生成するトピックファイル
```

**重要な動作仕様**:
- `MEMORY.md` のみセッション開始時にロード（上限: 200行 または 25KB）
- トピックファイルはオンデマンド（Claude が必要と判断した時のみ Read ツールで取得）
- 同一 git リポジトリの全 worktree でメモリディレクトリを共有
- マシンローカル（クラウドや別マシンには同期しない）
- サブエージェントも独自の auto memory を持てる

**設定オプション**:
```json
// ~/.claude/settings.json
{
  "autoMemoryEnabled": false,          // 無効化（または CLAUDE_CODE_DISABLE_AUTO_MEMORY=1）
  "autoMemoryDirectory": "~/my-memory" // カスタム保存先（project settings では設定不可）
}
```

**`/memory` コマンド**: 現在ロードされている全 CLAUDE.md・rules ファイル一覧表示 + auto memory の ON/OFF トグル + メモリフォルダへのリンクを提供。

**適用提案**:
```bash
# セッション内で「覚えておいて」と伝えると auto memory に保存
"always use pnpm, not npm を覚えておいて"
# → ~/.claude/projects/<project>/memory/ 以下に保存される

# 手動で確認・編集
/memory
# → メモリファイルを開いて確認・削除
```

### CLAUDE.md: HTML コメントによるトークン節約

**発見内容**: CLAUDE.md 内の HTML ブロックコメント（`<!-- -->`）はコンテキスト注入前に自動で除去される。保守担当者向けの注釈をコンテキストを消費せず書ける。

```markdown
<!-- このセクションは 2026-03 に API 仕様変更により追加。変更時は api-team に連絡 -->
## API 設計規約
- エラーレスポンスは RFC 7807 形式を使用
- 全エンドポイントに OpenAPI コメントを追加
```

**制限**: コードブロック内のコメントは除去されない（通常通り保持される）。  
`/memory` でファイルを直接 Read した場合はコメントが見える。

**適用提案**: CLAUDE.md の各セクションに保守用メモ（追加理由・担当者・変更履歴）を HTML コメントで記載し、トークン消費ゼロで管理する。

### hooks: `if` フィールドによるツール名+引数の細粒度フィルタリング

**発見内容**: v2.1.85 以降、hooks に `if` フィールドが追加。`matcher`（ツール名のみ）に加えて**ツール名 + 引数**の組み合わせでフィルタリングできる。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "if": "Bash(git *)",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/check-git-policy.sh"
          },
          {
            "type": "command",
            "if": "Bash(rm *)",
            "command": "echo '削除コマンドを検出' >&2 && exit 2"
          }
        ]
      }
    ]
  }
}
```

**`if` フィールドの特徴**:
- パーミッションルールと同じ構文: `Bash(git *)`, `Edit(*.ts)`, `Write(.env*)`
- `matcher` がグループ全体のフィルタ、`if` が個別ハンドラのフィルタ（二段階）
- **適用可能なイベント**: `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PermissionRequest`, `PermissionDenied` のみ
- 非対応イベントに設定すると hook が実行されなくなる

**適用提案**: Bash hook の `if` でコマンド別に分岐させ、`git` コマンドのみポリシーチェック、`rm` コマンドのみ確認プロンプト、という細粒度制御を実現。

### hooks: Stop フックの無限ループ防止（`stop_hook_active`）

**発見内容**: `Stop` フックで Claude が作業を継続するよう指示すると、次の Stop フックが再び実行され無限ループに陥る。`stop_hook_active` フィールドで対処。

```bash
#!/bin/bash
# ~/.claude/hooks/verify-complete.sh
INPUT=$(cat)

# 無限ループ防止: Stop フック自体によって発火した場合はスキップ
if [ "$(echo "$INPUT" | jq -r '.stop_hook_active')" = "true" ]; then
  exit 0  # Claude の終了を許可
fi

# テストが通っているか検証
if ! npm test --silent 2>/dev/null; then
  echo "テストが失敗しています。修正してから完了してください。" >&2
  exit 1  # Claude に作業継続を指示
fi

exit 0
```

**適用提案**: Stop フックでテスト実行や完了確認を行う場合は必ず `stop_hook_active` チェックを先頭に追加。

### hooks: `InstructionsLoaded` フック（デバッグ用）

**発見内容**: `InstructionsLoaded` フックが CLAUDE.md や `.claude/rules/*.md` がコンテキストにロードされるたびに発火。パス別ルールのデバッグに活用できる。

```json
{
  "hooks": {
    "InstructionsLoaded": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "jq -c '{file: .file_path, reason: .load_reason, timestamp: now | todate}' >> ~/claude-instructions-log.txt"
          }
        ]
      }
    ]
  }
}
```

**matcher（`load_reason`）の値**:
- `session_start`: セッション開始時
- `nested_traversal`: サブディレクトリ探索時
- `path_glob_match`: paths フロントマターのパターン一致
- `include`: `@import` 参照
- `compact`: コンパクション後の再注入

**適用提案**: 「なぜこのルールが効かないのか」を調査する際に `InstructionsLoaded` ログで実際のロードタイミングを確認。

---

### hooks: `CLAUDE_ENV_FILE` による環境変数永続化

**発見内容**: `CwdChanged` および `FileChanged` フックのコマンドが `$CLAUDE_ENV_FILE` に書き込むと、Claude の次の Bash コマンド実行前にその環境変数が適用される。

```json
{
  "hooks": {
    "CwdChanged": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "direnv export bash >> \"$CLAUDE_ENV_FILE\""
          }
        ]
      }
    ],
    "FileChanged": [
      {
        "matcher": ".envrc|.env",
        "hooks": [
          {
            "type": "command",
            "command": "direnv export bash >> \"$CLAUDE_ENV_FILE\""
          }
        ]
      }
    ]
  }
}
```

**活用シナリオ**: direnv を使うプロジェクトで、Claude が `cd` するたびに正しい環境変数が適用される。Python の virtualenv 切り替え、AWS プロファイル切り替えにも応用可能。
