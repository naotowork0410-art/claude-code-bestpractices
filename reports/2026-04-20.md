## 今週の新発見

### CLAUDE.md: `/init` コマンドと精度重視の記述原則

**発見内容**: 公式ドキュメントとコミュニティ記事で「精度（Precision）がリビジョン数を削減する」という原則が繰り返し強調された。また `/init` コマンドによる自動生成が新たに推奨手順として明確化。

- **`/init` コマンド**: プロジェクトディレクトリで実行すると、ディレクトリ構造・技術スタックを検出して starter CLAUDE.md を自動生成。ゼロから書く必要がなくなる。
- **具体的参照の原則**: ファイルパス・制約・サンプルパターンを明示するほどリビジョンが減る。「TypeScript を使う」より「`src/api/**/*.ts` のファイルは必ず Zod でバリデーションを行う」のように記述。
- **10-section 構造**: Medium 記事で効果が検証された推奨構造:
  1. Project Context（1行サマリー）
  2. Architecture Overview（ディレクトリ構成）
  3. Tech Stack（バージョン含む）
  4. Code Standards（型ヒント・命名規則等）
  5. Key Commands（test/build/lint 正確なコマンド）
  6. Environment Setup（依存関係の注意点）
  7. Gotchas（既知の罠・回避策）
  8. Compaction Instructions（コンパクション後に保持する情報）
  9. Anti-patterns（禁止パターン）
  10. References（重要ドキュメントへのリンク）

**適用提案**:
```bash
# 既存プロジェクトに即座に適用
cd /path/to/project
claude
/init
# → 生成されたドラフトを上記10-section 構造に沿って補強する
```

### hooks: 3つのカデンス分類と実行順の設計原則

**発見内容**: hooks のイベントは「いつ発火するか」で3カデンスに分類されており、それぞれ設計指針が異なることが公式ドキュメントで明確化された（前回レポートはイベント名の列挙のみ）。

| カデンス | イベント | 設計指針 |
|---------|---------|---------|
| **セッションあたり1回** | `SessionStart`, `SessionEnd` | セットアップ・クリーンアップ処理。重い初期化はここへ |
| **ターンあたり1回** | `UserPromptSubmit`, `Stop`, `StopFailure` | ターン完了後の検証。テストスイート・ビルドはここ |
| **ツール呼び出しごと** | `PreToolUse`, `PostToolUse` | 軽量処理のみ。重い処理は`Stop`へ移行を推奨 |

**新たに判明した実装パターン**:

1. **PreToolUse でのブロック理由を stderr に書く**:
   ```bash
   #!/bin/bash
   # PreToolUse: 機密ファイルへの書き込みをブロック
   if echo "$CLAUDE_TOOL_INPUT" | grep -q "\.env\|secrets/"; then
     echo "機密ファイルへの書き込みはブロックされました: $CLAUDE_TOOL_INPUT" >&2
     exit 2  # exit 2 = ブロック。理由は stderr 経由で Claude に伝達
   fi
   ```

2. **PostToolUse の `; exit 0` フォールバック**:
   ```bash
   #!/bin/bash
   # PostToolUse: 自動フォーマット（失敗してもワークフローをブロックしない）
   prettier --write "$CLAUDE_TOOL_OUTPUT_PATH" 2>/dev/null; exit 0
   ```

3. **カデンス別の責務分離パターン**:
   ```json
   {
     "hooks": {
       "PreToolUse": [
         {"matcher": "Write|Edit", "command": "~/.claude/hooks/scan-secrets.sh"}
       ],
       "PostToolUse": [
         {"matcher": "Write|Edit", "command": "prettier --write $FILE; exit 0"}
       ],
       "Stop": [
         {"command": "npm test && npm run lint || true"}
       ],
       "SessionStart": [
         {"command": "~/.claude/hooks/restore-context.sh"}
       ]
     }
   }
   ```

**適用提案**: 既存の PostToolUse にテストスイートを置いている場合、`Stop` フックに移行してツール呼び出しのたびに実行されるオーバーヘッドを排除する。

### settings.json: モデル制御と `/status` デバッグコマンド

**発見内容**: `settings.json` のモデル制御項目が充実しており、`/status` コマンドでどの設定ファイルが有効かを確認できることが判明。

**新規設定項目**:
```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "model": "claude-opus-4-7",
  "effortLevel": "high",
  "alwaysThinkingEnabled": true,
  "availableModels": ["claude-opus-4-7", "claude-sonnet-4-6"],
  "permissions": {
    "deny": [
      "Write(.env*)",
      "Write(secrets/**)",
      "Bash(rm -rf *)"
    ]
  },
  "env": {
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "65"
  }
}
```

**`/status` コマンドの活用**:
- 「設定が効いていない」問題の原因特定に必須
- どのスコープ（user/project/local）の設定が優先されているかを表示
- 設定の競合（プロジェクト設定がユーザー設定を上書きしている等）を可視化

**スコープ優先度**（高い順）:
1. `.claude/settings.local.json`（個人・非コミット）
2. `.claude/settings.json`（チーム・コミット）
3. `~/.claude/settings.json`（グローバル）

**適用提案**:
```bash
# 設定が反映されない場合のデバッグ手順
/status
# → Active settings sources を確認し、競合するファイルを特定
```

### コンテキスト管理: 定量的劣化モデルと「アーキテクチャ優先」原則

**発見内容**: コンテキスト劣化についての定量データと、2026年における「プロンプト技巧よりコンテキスト設計が重要」という原則が複数ソースで確認された。

**コンテキスト劣化の定量データ**:
| 使用率 | 状態 |
|-------|------|
| 0〜20% | 最適パフォーマンス |
| 20〜40% | 劣化開始（軽微） |
| 60%超 | 指示の見落とし・エラー頻発 |
| 90% | 深刻な劣化。以前の指示を「忘れる」 |

**2026年の重要洞察**:
- Anthropic 内部テストで、ガイダンスなしの実装成功率は **33%** 。CLAUDE.md・hooks・Plan Mode を組み合わせると大幅改善。
- 「プロンプトの書き方の巧みさ」より「コンテキストの構造設計（CLAUDE.md、MCP Tool Search、サブエージェント分離）」の方が安定した出力に直結する。

**適用提案**:
```markdown
<!-- ~/.claude/CLAUDE.md に追加 -->
## セッション管理ルール
- 1セッション = 1タスク原則。無関係なタスクは /clear 後に開始。
- コンテキスト60%超でステータスラインが警告 → 即座に /compact を実行。
- 2回連続で同じ修正が失敗した場合は /clear してプロンプトを再設計。
```

### スラッシュコマンド / Skills 統合（2026年アップデート）

**発見内容**: 2026年版で `.claude/commands/`（旧スラッシュコマンド）と `.claude/skills/`（Skills）が**統合**された。既存の commands/ ファイルは後方互換で動作するが、今後は skills/ に一本化が推奨。

**統合後の推奨ファイル構造**:
```
.claude/
├── skills/
│   ├── ship/
│   │   └── SKILL.md   # /ship コマンドに対応
│   ├── review/
│   │   └── SKILL.md   # /review コマンドに対応
│   └── debug/
│       └── SKILL.md
└── commands/           # 後方互換（新規作成は非推奨）
    └── old-command.md
```

**実践的なスキルテンプレート**:
```yaml
# .claude/skills/ship/SKILL.md
---
name: ship
description: Review diff, run tests, commit and push to main
allowed-tools: Bash(git *) Bash(npm test) Bash(npm run lint)
effort: medium
disable-model-invocation: true  # 手動呼び出しのみ（自動デプロイ防止）
---

現在の差分をレビューしてください:
1. `npm test` を実行してテストが全て通ることを確認
2. `npm run lint` でリントチェック
3. 問題がなければ明確なコミットメッセージで `git commit` して `git push`

問題があれば修正してから再実行してください。
```

**コミュニティ製プロダクション対応コレクション**:
- `wshobson/commands`: 本番対応のスラッシュコマンド集
- `hesreallyhim/awesome-claude-code`: Skills・hooks・MCPの厳選リスト
- `travisvn/awesome-claude-skills`: Skills 専門のキュレーション

**適用提案**: `wshobson/commands` を参考に `/ship`・`/review`・`/debug` の3コマンドを優先整備する。

### MCP: `managed-mcp.json` によるエンタープライズ配布

**発見内容**: チーム/エンタープライズ環境向けに `managed-mcp.json` でサーバー設定を固定配布できることが判明。ユーザーによるカスタマイズを許可しつつポリシー制御を維持できる。

**設定の使い分け**:
| ファイル | 用途 | カスタマイズ |
|---------|------|------------|
| `.mcp.json` | プロジェクト共有（git 管理） | チームメンバーが変更可 |
| `managed-mcp.json` | 組織ポリシー固定 | ユーザーは追加のみ可（既存を変更不可） |
| `~/.claude/mcp.json` | 個人グローバル設定 | 完全自由 |

**セキュリティベストプラクティス（最小権限原則）**:
```json
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN_READONLY}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres", "${DB_URL_READONLY}"]
    }
  }
}
```

**適用提案**: 
- `GITHUB_TOKEN` は `repo:read`・`issues:read` のみのスコープで発行し `${ENV_VAR}` 参照に。
- Tool Search 機能（MCP ツール定義を遅延ロード）により大量ツールがあってもコンテキストを 85〜95% 削減。Context7 MCP でドキュメント参照を強化する場合も同様。

### フィードバックループ埋め込みパターン

**発見内容**: Anthropic の推奨として「プロンプト自体にフィードバックループ（検証手段）を埋め込む」パターンが強調された。Claude が自律的に自分のミスを検出できる。

**パターン例**:
```markdown
次のタスクを実行してください: [タスク内容]

実装後、以下で自己検証してください:
1. `npm test` — 全テストが PASS すること
2. `npm run type-check` — 型エラーがゼロであること
3. `git diff --stat` — 変更ファイルが期待どおりであること

検証が失敗した場合は、エラーメッセージを解析して自律的に修正してください。
```

**CLAUDE.md への組み込み**:
```markdown
# 実装後の自己検証プロトコル
コード変更を完了したら必ず以下を実行すること:
- `make test` で単体テストを実行
- `make lint` でリントチェック
- スクリーンショットが必要な UI 変更は playwright でキャプチャ
```

**適用提案**: CLAUDE.md の「Key Commands」セクションに検証コマンドを記載し、Claude が実装後に自動実行するよう指示する。
