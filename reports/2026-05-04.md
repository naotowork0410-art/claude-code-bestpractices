# Claude Code ベスプラ調査レポート 2026-05-04

## 今週の新発見

### `.claude/rules/` によるパススコープ型ルール管理

**発見内容**: `.claude/rules/` ディレクトリを使うと、ルールをトピック別ファイルに分割しつつ、特定ファイルパターンにのみロードされる条件付きルールを定義できる。YAML フロントマターの `paths:` フィールドでグロブパターンを指定し、Claude が対象ファイルを読んだときだけ該当ルールがコンテキストに追加される（セッション開始時は未ロード）。これにより大きな CLAUDE.md の「肥大化問題」を構造的に解決できる。

**ディレクトリ構成例**:
```
.claude/
├── CLAUDE.md               # 全セッション共通の最小限ルール
└── rules/
    ├── api-design.md        # paths なし → 常時ロード
    ├── frontend.md          # paths: src/components/**/*.tsx
    ├── security.md          # paths: src/api/**
    └── testing.md           # paths: **/*.test.ts, **/*.spec.ts
```

**パススコープ rules の書き方**:
```markdown
---
paths:
  - "src/api/**/*.ts"
  - "src/api/**/*.tsx"
---

# API 開発ルール

- 全エンドポイントに入力バリデーションを含める
- 標準エラーレスポンス形式を使用する
- OpenAPI コメントを必ず記述する
```

**シンボリックリンクによる複数プロジェクト共有**:
```bash
# 共通ルールをシンボリックリンクで各プロジェクトに展開
ln -s ~/shared-claude-rules .claude/rules/shared
ln -s ~/company-standards/security.md .claude/rules/security.md
```

**ユーザーレベルルール**: `~/.claude/rules/` に置いたルールは全プロジェクトに適用される（プロジェクト設定より優先度低）。

**Skills との使い分け**:
| 用途 | 手段 |
|------|------|
| 常時 or ファイル種別ベースで適用 | `.claude/rules/*.md` |
| 呼び出し時のみ適用（副作用があるワークフロー） | `.claude/skills/<name>/SKILL.md` |

**適用提案**: `~/.claude/rules/preferences.md` を作成し、全プロジェクト共通の個人設定（コードスタイル・言語設定等）を記述する。プロジェクトの CLAUDE.md は 50 行以下に絞り、細粒度ルールはすべて `.claude/rules/` へ移行する。

### 検証ファースト原則 ― 単一最大レバレッジポイント

**発見内容**: Anthropic 公式ベストプラクティスが「Claude が自分の作業を検証できる手段を与えることが、単一の最大レバレッジポイント」と明記した。テスト・スクリーンショット・期待出力などの成功基準を提供すると、Claude のパフォーマンスは劇的に向上する。

**Before / After 比較表**（公式ドキュメントより）:

| 戦略 | Before（曖昧） | After（検証付き） |
|------|----------------|-------------------|
| 機能実装 | `"validateEmailを実装して"` | `"validateEmail関数を書いて。テストケース: user@example.com=true, invalid=false, user@.com=false。実装後にテストを実行して"` |
| UI 変更 | `"ダッシュボードをよく見せて"` | `"[スクリーンショット貼付] このデザインを実装して。結果のスクリーンショットを撮って元と比較し、差分をリストアップして修正して"` |
| バグ修正 | `"ビルドが失敗してる"` | `"ビルドがこのエラーで失敗: [エラー貼付]。修正してビルド成功を確認して。症状ではなく根本原因を対処して"` |

**適用提案**: 全タスクのプロンプトに「実装後に `npm test` を実行して確認」「結果が期待値と一致することを確認」などの検証指示を必ず付け加える。CI パイプラインなら `claude -p "..." && npm test` でワンライナー化する。

### AskUserQuestion インタビューパターン ― SPEC.md 生成ワークフロー

**発見内容**: 大型機能の実装前に「Claude に自分をインタビューさせる」パターンが公式推奨になった。Claude が `AskUserQuestion` ツールを使い、技術実装・UX・エッジケース・トレードオフについて深掘り質問を行い、最終的に `SPEC.md` を生成する。その後 **別の新しいセッション**で実装を開始することでコンテキストをクリーンに保つ。

**プロンプトテンプレート**:
```
私は [機能の簡単な説明] を構築したい。AskUserQuestion ツールを使って詳細なインタビューをしてください。

技術実装・UI/UX・エッジケース・懸念事項・トレードオフについて質問してください。
自明な質問はせず、私がまだ考慮していない難しい部分を掘り下げてください。

十分にカバーできたら、完全な仕様を SPEC.md に書いてください。
```

**ワークフロー**:
1. インタビューセッションで SPEC.md を生成（コンテキストが仕様議論で満たされる）
2. `/clear` または新セッション起動
3. `"SPEC.md に基づいて実装してください"` で実装開始（クリーンなコンテキスト）

**適用提案**: 中・大規模機能の実装前に必ずインタビューセッションを挟む。SPEC.md はプロジェクトルートへコミットし、チームの仕様書として活用する。

### CLAUDE.md 組織管理 ― 新機能まとめ

**発見内容**: 組織・チーム規模での CLAUDE.md 管理に関する複数の新機能が確認された。

#### 1. 組織全体のマネージドポリシー CLAUDE.md

管理者が全ユーザーに適用される CLAUDE.md をシステムパスに配置できる。**個人設定では除外不可**（最高優先度）。

| OS | パス |
|----|------|
| macOS | `/Library/Application Support/ClaudeCode/CLAUDE.md` |
| Linux / WSL | `/etc/claude-code/CLAUDE.md` |
| Windows | `C:\Program Files\ClaudeCode\CLAUDE.md` |

**managed settings との役割分担**:
| 用途 | 設定場所 |
|------|---------|
| ツール・コマンドのブロック、サンドボックス強制 | `managed settings: permissions.deny / sandbox` |
| 環境変数・API プロバイダルーティング | `managed settings: env` |
| コーディング規約・コンプライアンス案内 | Managed CLAUDE.md |

#### 2. AGENTS.md クロスツール互換

他のコーディングエージェント（OpenHands 等）が使う `AGENTS.md` が既に存在するプロジェクトで、重複なく Claude にも読ませる方法:
```markdown
<!-- CLAUDE.md -->
@AGENTS.md

## Claude Code 固有設定

`src/billing/` の変更は plan mode を使用すること。
```

#### 3. `--add-dir` + 追加ディレクトリの CLAUDE.md ロード

```bash
# 作業ディレクトリ外の共有設定を Claude に読み込ませる
CLAUDE_CODE_ADDITIONAL_DIRECTORIES_CLAUDE_MD=1 claude --add-dir ../shared-config
```

`.claude/rules/*.md` と `CLAUDE.local.md` も追加ディレクトリから読み込まれる。

#### 4. `/compact` 後のサバイバル動作（重要）

| ファイル | `/compact` 後の動作 |
|---------|-------------------|
| プロジェクトルート `CLAUDE.md` | **自動再読み込み**（ディスクから再注入） |
| サブディレクトリの `CLAUDE.md` | **自動再読み込みされない**（次回対象ファイルアクセス時に再ロード） |
| 会話内で直接伝えた指示 | **消失**（CLAUDE.md に書かないと失われる） |

**適用提案**: 重要な指示は必ず CLAUDE.md に書く。`/compact` 後に設定が消えた場合は `/memory` で確認する。

### `/memory` コマンドと Auto Memory 詳細

**発見内容**: `/memory` コマンドが公式に推奨されるメモリ管理インターフェースとして確立した。セッション内で読み込まれている全ファイルを一覧でき、Auto Memory のオン/オフも切り替えられる。

**`/memory` でできること**:
- 読み込まれている CLAUDE.md・CLAUDE.local.md・rules の一覧確認
- Auto Memory のオン/オフ切り替え
- Auto Memory フォルダをエディタで開く
- `"pnpm を常に使用して"` のような口頭指示を Claude に Auto Memory へ保存させる

**Auto Memory 設定オプション**:
```json
// ~/.claude/settings.json
{
  "autoMemoryEnabled": true,
  "autoMemoryDirectory": "~/my-memory-dir"
}
```
```bash
# 環境変数でも無効化可能
CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 claude
```

**ストレージ構造**:
```
~/.claude/projects/<project>/memory/
├── MEMORY.md          # 毎セッション先頭200行 or 25KB をロード（インデックス）
├── debugging.md       # オンデマンドで読まれる詳細ノート
└── api-conventions.md # トピックファイル
```

**適用提案**: セッション中に「これを覚えておいて」と伝えると Auto Memory に書かれる。定期的に `/memory` で内容を確認・整理し、不要なエントリを削除して肥大化を防ぐ。

### Claude in Chrome 拡張 ― UI 自動検証

**発見内容**: Claude Code の Chrome 拡張が公式ドキュメントに UI 検証ツールとして明記された。Claude がブラウザで新しいタブを開き、UI をテストし、スクリーンショットを撮影して比較し、コードが動作するまで繰り返し修正する。

**活用パターン**:
```
# フロントエンド作業時のプロンプト例
"このデザイン [スクリーンショット] を実装して。
Chrome 拡張を使って結果を確認し、元のデザインと比較して差分を修正してください。"
```

**適用提案**: フロントエンド変更時は「実装後にスクリーンショットで確認して差分を修正」をプロンプトに追加するだけで、Claude が自律的に視覚的検証を行う。`/permissions` で頻繁に使うドメインを許可リストに追加しておく。

### `--append-system-prompt` ― 自動化での確実な指示注入

**発見内容**: `--append-system-prompt` フラグを使うと、CLAUDE.md ではなく **システムプロンプトレベル**で指示を注入できる。CLAUDE.md はユーザーメッセージとして届くが、このフラグを使った指示はシステムプロンプトに追加される。非インタラクティブ自動化スクリプトで最重要な制約を守らせる場合に有用。

```bash
# CI パイプラインで特定の制約を確実に適用
claude -p "全テストを修正して" \
  --append-system-prompt "本番ファイルへの変更は禁止。テストファイルのみ変更すること" \
  --output-format json
```

**注意**: 毎回の呼び出しで渡す必要があるため、インタラクティブセッションより **スクリプト・自動化向け**。

**適用提案**: CI/CD での `claude -p` 呼び出しに重要な制約を `--append-system-prompt` で渡し、CLAUDE.md に頼らない強制力のある指示を適用する。
