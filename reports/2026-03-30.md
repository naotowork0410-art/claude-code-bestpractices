# Claude Code ベスプラ調査レポート 2026-03-30

## 今週の新発見

### CLAUDE.md設定

**発見内容**: CLAUDE.mdは階層的に読み込まれる。`~/.claude/CLAUDE.md`（全プロジェクト共通）→ プロジェクトルートの`./CLAUDE.md`（チーム共有）→ サブディレクトリの`CLAUDE.md`（必要時に読み込み）。また `.claude/rules/*.md` でルールをモジュール分割できる。`@path/to/file` 構文で他ファイルをインポート可能。

**適用提案**:
- `~/.claude/CLAUDE.md` を作成し、全プロジェクト共通の個人スタイル・ツール設定を記述
- プロジェクトルートの `CLAUDE.md` には「チームが共有すべきルール」のみ記載しgit管理
- `.claude/rules/security.md` / `.claude/rules/frontend.md` 等に分割してマージ競合を回避
- `IMPORTANT:` や `YOU MUST` で厳守ルールを強調
- `/init` コマンドで初期CLAUDE.mdを自動生成し、不要部分を削除する

**CLAUDE.md推奨構成**:
```markdown
# Project Context
（プロジェクトの概要・技術スタック）

## Development Commands
- テスト実行コマンド
- リント・ビルドコマンド

## Code Style
（デフォルトと異なるルールのみ記載）

## Key Directories
（主要ディレクトリの役割）

## Gotchas
（非自明な注意事項・落とし穴）
```

**アンチパターン**:
- CLAUDE.mdを肥大化させない（重要なルールが埋もれる）
- フォーマット強制にCLAUDE.mdを使わない（リンター/フォーマッターを使う）
- 「きれいなコードを書く」等の自明なルールは書かない

### settings.json

**発見内容**: 設定ファイルは5段階の優先順位がある（managed > CLI引数 > settings.local.json > settings.json > グローバル）。`$schema` 指定でVS Codeの補完が効く。`CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` で自動コンパクションの閾値を調整できる。`permissions.allow/deny` でツール呼び出しの粒度制御が可能。

**適用提案**:
- `.claude/settings.json` をgit管理（チーム共有のpermissions/hooksを記載）
- `.claude/settings.local.json` に個人設定（auto-gitignored）
- `~/.claude/settings.json` にグローバル個人設定

```json
{
  "$schema": "https://json.schemastore.org/claude-code-settings.json",
  "permissions": {
    "allow": [
      "Bash(npm run lint)",
      "Bash(npm run test *)",
      "Bash(git *)"
    ],
    "deny": [
      "Read(./.env)",
      "Read(./.env.*)",
      "Bash(rm -rf *)"
    ]
  },
  "env": {
    "CLAUDE_AUTOCOMPACT_PCT_OVERRIDE": "75"
  },
  "cleanupPeriodDays": 14,
  "includeCoAuthoredBy": true
}
```

- `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE: "75"` を設定してコンテキスト劣化前にコンパクション
- `/status` コマンドで設定の読み込み元を確認、`/config` でUI表示

### hooks（PreToolUse / PostToolUse / SessionStart 等）

**発見内容**: hooksは18種類のイベントをサポート。コマンド(command)・プロンプト(prompt)・エージェント(agent)・HTTP(http) の4タイプがある。**exit code 2** でアクションをブロックしClaudeにフィードバック（exit 1は警告ログのみ）。hooksはサブエージェントにも再帰的に適用される。

**適用提案**:

1. **PostToolUse: 自動フォーマット**
```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{
        "type": "command",
        "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
      }]
    }]
  }
}
```

2. **PreToolUse: 機密ファイル保護**
```bash
#!/bin/bash
# .claude/hooks/protect-files.sh
INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
PROTECTED=(".env" "secrets/")
for pattern in "${PROTECTED[@]}"; do
  if [[ "$FILE_PATH" == *"$pattern"* ]]; then
    echo "Blocked: $FILE_PATH は保護対象ファイルです" >&2
    exit 2
  fi
done
exit 0
```

3. **SessionStart: コンパクション後のコンテキスト復元**
```json
{
  "hooks": {
    "SessionStart": [{
      "matcher": "compact",
      "hooks": [{
        "type": "command",
        "command": "echo 'リマインダー: npmではなくBunを使用。コミット前に bun test を実行。'"
      }]
    }]
  }
}
```

4. **Notification: デスクトップ通知（Linux）**
```json
{
  "hooks": {
    "Notification": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "notify-send 'Claude Code' 'Needs attention'"
      }]
    }]
  }
}
```

**注意点**:
- hooksは200ms以内に終了させる
- `~/.zshrc` に無条件の `echo` がある場合、JSON出力が壊れる（インタラクティブシェル判定で回避）
- Stop hookは `stop_hook_active` を確認して無限ループを防ぐ

### MCP接続

**発見内容**: MCPサーバーの設定スコープは4段階（managed > ユーザー個人 > プロジェクト `.mcp.json` > グローバル設定）。トランスポートは **stdio（ローカル）** と **HTTP（クラウド）** が推奨。ツール定義がコンテキストを圧迫するため、Tool Search機能で動的ロードすると最大85%削減可能。

**適用提案**:
- `.mcp.json` をプロジェクトルートに配置してgit管理
- シークレットは環境変数展開 `${GITHUB_TOKEN}` で参照（直接ハードコードしない）

```json
{
  "mcpServers": {
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    },
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@playwright/mcp@latest"]
    }
  }
}
```

- `enableAllProjectMcpServers: true` で毎回の承認をスキップ（信頼済みプロジェクトのみ）
- `claude mcp list` で接続状態を確認
- Node.js 18以上が必須（最頻出エラーの原因）

### カスタムスラッシュコマンド（Skills）

**発見内容**: `.claude/skills/<name>/SKILL.md` が新しい推奨形式（レガシーは `.claude/commands/<name>.md`）。フロントマターで `disable-model-invocation: true`（副作用あるコマンドをユーザー専用に制限）、`context: fork`（独立した実行コンテキスト）、`allowed-tools`（ツール制限）が指定可能。`!` プレフィックスでシェルコマンド出力を動的にインジェクトできる。

**適用提案**:
```yaml
# .claude/skills/fix-issue/SKILL.md
---
name: fix-issue
description: GitHubイシューを調査・修正・PR作成まで自動実行する
argument-hint: "[issue-number]"
disable-model-invocation: true
allowed-tools: Bash(gh *), Read, Grep, Edit
context: fork
---

GitHub issue $ARGUMENTS を修正してください:
1. `gh issue view $ARGUMENTS` でイシュー詳細を確認
2. 関連ファイルを特定して修正を実装
3. テスト実行
4. コミット・PR作成: `gh pr create`
```

```yaml
# .claude/skills/pr-summary/SKILL.md
---
name: pr-summary
allowed-tools: Bash(gh *)
---

## PR情報（自動取得）
- 差分: !`gh pr diff`
- コメント: !`gh pr view --comments`
- 変更ファイル: !`gh pr diff --name-only`

このPRのビジネスインパクトを要約してください。
```

**組み込みスキル活用**:
- `/batch <instruction>` — 大規模変更を並列処理（独立したgit worktreeで実行）
- `/simplify` — 3つの並列レビューエージェントでコード品質改善
- `/loop 5m <prompt>` — デプロイ監視等の定期実行

### メモリ管理・コンテキスト節約

**発見内容**: コンテキストが蓄積するとエラー率上昇・指示忘れが発生。主な管理手法: `/clear`（タスク切り替え時）、`/compact`（手動コンパクション）、`/rewind`（チェックポイント復元）、`/btw`（サイドクエスチョン、コンテキスト非追加）。サブエージェントは独立したコンテキストウィンドウで動作するため大規模調査に有効。

**適用提案**:
- `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE: "75"` で早めにコンパクション（95%デフォルトより安全）
- タスク切り替え時は必ず `/clear` を実行（「キッチンシンクセッション」アンチパターンを回避）
- 修正が2回失敗したら `/clear` してより良いプロンプトで再スタート
- CLAUDE.mdにコンパクション時の保持指示を記載:
  ```markdown
  When compacting, always preserve:
  - 変更済みファイルの一覧
  - 実行したテストコマンド
  - 現在のタスク説明
  ```
- サブエージェントで大規模コードベース調査を分離（本体コンテキストを保護）

### エージェント・サブエージェント

**発見内容**: カスタムサブエージェントを `.claude/agents/<name>.md` に定義できる。`memory: project` でセッション間の知識を `.claude/agent-memory/<name>/MEMORY.md` に永続化。`model: haiku` でコスト削減（調査系）、`model: opus` で高精度推論（レビュー系）。`isolation: worktree` で安全な実験的変更が可能。

**適用提案**:
```markdown
---
name: security-reviewer
description: 認証・データ処理変更後に自動的にセキュリティレビューを実施
tools: Read, Grep, Glob
model: opus
memory: project
---

セキュリティエンジニアとしてコードをレビューしてください:
- インジェクション脆弱性（SQL, XSS, コマンドインジェクション）
- 認証・認可の欠陥
- シークレットのハードコード
- 安全でないデータ処理

各問題について具体的な行番号と修正案を提示してください。
```

**Writer/Reviewerパターン**:
- セッションA: 実装（コンテキストあり）
- セッションB: レビュー（新鮮なコンテキスト、偏りなし）
- BのレビューをAにフィードバック

**Fan-Outパターン**（大規模並列処理）:
```bash
for file in $(cat files.txt); do
  claude -p "Migrate $file to async" --allowedTools "Edit,Bash(git commit *)" &
done
wait
```
